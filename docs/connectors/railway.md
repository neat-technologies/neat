# Railway connector

Second implementation of the [connectors plane](./README.md) (ADR-124, ADR-127). Pulls
Railway's own edge/ingress telemetry and mints OBSERVED edges fused onto the `RouteNode`s
`packages/core/src/extract/routes.ts` already builds — no app instrumentation required, and
no client SDK to recognize in the first place, since a Railway-hosted app's own HTTP routes
simply run on Railway's infrastructure.

## Scope

- **Target: Railway's GraphQL API only.** Railway is SaaS-only — there is no self-hosted
  Railway product, so unlike Supabase there is no Cloud-vs-self-hosted branch to resolve
  and none expected later.
- **Hosting-platform shape, not client-SDK shape.** The Supabase connector fuses onto
  `supabase-js` call sites because the app imports a client library that names a table or
  RPC. Railway names nothing in application code at all — the signal comes entirely from
  Railway's own edge/ingress layer watching traffic hit routes the app already declares.
  Fusion therefore binds onto the `RouteNode` static extraction already produces, the same
  way `route-match.ts` (ADR-119) binds an HTTP client call site onto a server route — this
  connector is the first to reuse that pattern from the connectors-plane side rather than
  the static-extraction side.
- **Both credential profiles ship day one.** Railway's `Project-Access-Token` is naturally
  environment-scoped (a token is minted per Railway environment, not per account), so it is
  already a real least-privilege credential unit for both local and hosted profiles — no
  Fork-A-style gap where the hosted profile has to ship a narrower cut first (`supabase.md`
  §Scope's log-surface-only sequencing has no equivalent here).

## Surfaces used

### 1. `httpLogs` (both profiles) — route-grain signal

Railway's GraphQL API (`backboard.railway.com/graphql/v2`) exposes `httpLogs`: a structured
per-request record generated by Railway's own edge/ingress layer, independent of whatever the
app writes to stdout. Each record carries `method`, `path`, `httpStatus`, `totalDuration`,
`requestId`, `edgeRegion`, and `deploymentId` — enough to reconstruct one HTTP call's shape
without any app-side change. **Needs-endpoint-testing:** the exact query field names, pagination
shape, and any lookback cap on this query are not yet confirmed against a live project; the
poller should be built against a real Railway project before locking in a poll implementation,
the same way the Supabase connector confirmed its rate-limit and scope assumptions against a
live endpoint rather than the docs alone.

- **Auth (both profiles):** a `Project-Access-Token`, scoped to one Railway project/environment.
  Both local and hosted profiles use the same token shape — see §Scope above.
- **Poll cadence:** on daemon tick / `neat sync` for local; a fixed interval for hosted,
  candidate value needs-endpoint-testing against whatever rate limit the GraphQL API enforces
  for this query (railway.com/docs does not appear to publish one as of this writing).

### 2. `httpMetrics` / `httpDurationMetrics` (both profiles) — pre-aggregated percentiles

The same GraphQL API exposes pre-aggregated request-count and latency-percentile time series,
filterable by exact path/method/status. This is an exact-string match, not a template — a
route with a path parameter (`/orders/:id`) does not collapse its instances into one series
here the way `RouteNode`'s own `pathTemplate` does, so this surface is a supplementary
volume/latency signal alongside `httpLogs` rather than a route-identity source in its own
right. `httpLogs` remains the surface this connector maps onto `RouteNode` identity; these
metrics enrich an already-resolved edge with pre-computed percentiles rather than requiring
NEAT to compute them from raw log rows itself — needs-endpoint-testing to confirm the exact
metric names and filter shape before wiring this in.

### 3. `networkFlowLogs` (both profiles) — service-dependency signal, independent of HTTP

L4 flow records carrying a `peerServiceId` field: which other Railway service a given service
exchanged network traffic with, at the connection level, with no HTTP request/response
framing at all. This is a distinguishing surface — no other connectors-plane provider
surveyed so far (Supabase) offers a signal that establishes a service dependency independent
of any request/RPC contract. See §Fusion below for how this mints an edge.

## Fusion — node identity

No new `NodeType`. Railway is the first connectors-plane provider that needs none, because the
`RouteNode` `routes.ts` already extracts (ADR-119) is exactly the node a per-request
`httpLogs` record should land on — the same node an OBSERVED server span would fuse onto if
the app were OTel-instrumented, and the same node an HTTP client call site resolves against
today.

**Route-grain fusion (`httpLogs` → `RouteNode`).** The connector normalizes a record's `path`
using the same path-normalization `route-match.ts` already applies when matching a client call
site against a server's declared route template. When normalization resolves a `(method,
pathTemplate)` pair to a `RouteNode` a static extractor already built, the connector mints a
file-grained OBSERVED `CALLS` edge onto it — reconciled onto the EXTRACTED service-relative
path the same way every other OBSERVED surface in NEAT lands file-grained (file-awareness.md
§4, ADR-118). When no `RouteNode` resolves — the app's framework/router isn't one
`routes.ts` recognizes yet, or the path doesn't match any declared template — the edge lands
service-level, honestly: the same missing-extracted divergence surfacing that every other
connectors-plane and OTel-ingest surface already produces when static coverage falls short of
observed traffic.

**Service identity mapping.** Railway names a service by its own GraphQL `serviceId`, which
will not generally match NEAT's manifest-derived `serviceId(name)` (`packages/types/src/identity.ts`)
— the two are different naming authorities with no shared source of truth. Unlike
`infraId`-style sub-resource identity (the pattern `supabase.md` extends for tables and RPCs),
Railway needs no new id shape at all: every node this connector targets is already a
`RouteNode` or a `ServiceNode` NEAT's existing identity helpers construct. What it does need is
an explicit config-time mapping from a Railway `serviceId` to the NEAT service name that
resolves `serviceId(name)` — supplied once at connector setup, the same "resolved once, never
guessed" discipline ADR-127 states for this mapping, not inferred from Railway API responses at
poll time.

**`networkFlowLogs` → OBSERVED `CONNECTS_TO`.** A flow record's `peerServiceId` pair, both
resolved to NEAT service names via the mapping above, mints an OBSERVED `CONNECTS_TO` edge
between the two `ServiceNode`s — independent of whether any route resolves for the traffic
that produced the flow. This is a new fusion pattern for `CONNECTS_TO`: every prior OBSERVED
`CONNECTS_TO` edge (ADR-118, in-process database reads; the queue and WebSocket edges built on
the same primitive) comes from a span — something an OTel SDK captured at a call site. This is
the first `CONNECTS_TO` minted from a connector's own platform-level signal with no span
involved anywhere in its path, and no other connectors-plane provider surveyed so far exposes
an equivalent. Whether this generalizes into a third `connectors.md` fusion pattern (alongside
route-fusion and client-SDK-fusion) is worth watching as more hosting-platform providers get
surveyed, rather than something this design commits to yet.

## Static extractor gap this connector exposes

None, in the sense `supabase.md` documents one: `routes.ts` already parses the route shapes
this connector needs (Express, Fastify, Next.js), so there is no missing call-site parser to
grow before route-grain fusion works. The gap that does exist is `routes.ts`'s own framework
coverage — a Railway-hosted app built on a router `routes.ts` doesn't yet recognize gets
service-level-only edges from this connector, the same honest fallback any unrecognized
framework already produces for OTel-derived route fusion today. That gap belongs to
`routes.ts`, not to this connector, and shrinks as `routes.ts` grows new framework support
independent of anything Railway-specific.

## Out of scope for this cut

Native OTel/APM ingestion — Railway's own blog states plainly that Railway does not collect
traces natively; the only path to a trace is the app bringing its own exporter, which is an
app-side OTLP push NEAT already ingests today and not a connectors-plane concern. Any signal
gated behind a Railway plan tier beyond what a `Project-Access-Token` already grants
(needs-endpoint-testing — the survey did not turn up a tiering distinction on the GraphQL API
itself, but this should be confirmed against Railway's current pricing page before the poller
assumes uniform access). Deploy-time and build-log signal — `httpLogs`/`httpMetrics`/
`httpDurationMetrics`/`networkFlowLogs` are all runtime-traffic surfaces; build and deploy
telemetry is a different question this design does not take up.
