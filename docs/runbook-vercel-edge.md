# Runbook — deploy-and-observe: Vercel edge-runtime tracing (ADR-126)

Next.js apps on Vercel run two runtimes. Node-runtime routes get the standard
`@opentelemetry/sdk-node` init (`instrumentation.node.ts`); edge-runtime routes and
middleware run in a V8 isolate where the Node SDK cannot execute at all, so NEAT
instruments them through `@vercel/otel`'s `registerOTel()` in a generated
`instrumentation.edge.ts` (ADR-126, [framework-installers §7](contracts/framework-installers.md)).
This runbook is how you confirm, against a **real Vercel deployment**, that
edge-runtime spans actually reach a NEAT daemon and mint `OBSERVED` edges — the
half of a Vercel app that is dark by default.

Everything up to "Deploy to Vercel" you can do on a laptop. The deploy-and-observe
half needs a Vercel project and a NEAT daemon the deployment can reach over the
network.

## What the installer generates

`neat init --apply` on a Next.js package (`next` dep + `next.config.*`) writes a
three-file instrumentation shape at the package root (or under `src/` for
`create-next-app --src-dir` layouts):

- `instrumentation.ts` — the runtime gate. Next calls its `register()` once per
  runtime it boots; it dispatches on `process.env.NEXT_RUNTIME`:
  `'nodejs'` → `await import('./instrumentation.node')`,
  `'edge'` → `await import('./instrumentation.edge')`.
- `instrumentation.node.ts` — `@opentelemetry/sdk-node`, endpoint resolved from
  `neat-out/daemon.json` at boot (ADR-096).
- `instrumentation.edge.ts` — `registerOTel()` from `@vercel/otel`, wired to the
  **same** OTLP endpoint / protocol / bearer the Node file uses.

It also adds `@vercel/otel` to `package.json` `dependencies` (the one named
exception to the four-deps invariant, scoped to the edge file), and `.env.neat`
alongside. No application code changes beyond the generated files.

The generated `instrumentation.edge.ts` is exactly:

```ts
// Generated by `neat init --apply` (ADR-126). Next.js edge-runtime instrumentation via @vercel/otel.
import { registerOTel } from '@vercel/otel'

process.env.OTEL_SERVICE_NAME ||= '<service-name>'
process.env.OTEL_EXPORTER_OTLP_TRACES_ENDPOINT ||= 'http://localhost:4318/v1/traces'
process.env.OTEL_EXPORTER_OTLP_PROTOCOL ||= 'http/json'
if (process.env.NEAT_OTEL_TOKEN) process.env.OTEL_EXPORTER_OTLP_HEADERS ||= 'Authorization=Bearer ' + process.env.NEAT_OTEL_TOKEN

registerOTel({ serviceName: process.env.OTEL_SERVICE_NAME })
```

Why these four env lines and not a daemon.json walk like the Node file: the Edge
runtime forbids `node:fs`/`node:net`, so there is no filesystem probe to run —
the endpoint must arrive as an environment variable. `@vercel/otel@^2.x` reads
all four at `registerOTel()` time: it builds its OTLP exporter URL from
`OTEL_EXPORTER_OTLP_TRACES_ENDPOINT`, its wire format from
`OTEL_EXPORTER_OTLP_PROTOCOL` (`http/json` → the JSON exporter NEAT's receiver
validates hardest), and its auth header from `OTEL_EXPORTER_OTLP_HEADERS`. This
is the identical OTLP surface the Node installer wires — one receiver, one wire
format, one token.

## Prerequisites

- A Next.js app with at least one edge-runtime surface: a `middleware.ts` (always
  edge) and/or a route/handler declaring `export const runtime = 'edge'`. Middleware
  alone is enough — it is the most common edge surface and runs on every matched
  request.
- A Vercel account and the `vercel` CLI (`npm i -g vercel`), or a connected Git repo.
- A NEAT daemon reachable from Vercel's network. `localhost` is **not** reachable
  from a deployed isolate (see Troubleshooting) — you need a public/tunneled OTLP
  endpoint. For a quick loop, a tunnel (`cloudflared tunnel --url http://localhost:4318`,
  `ngrok http 4318`) in front of a local `neat` daemon works; for anything durable,
  run the daemon on a host with a routable address.

## 1. Generate and verify locally

From the app directory:

```bash
npx neat.is init . --apply          # or: neat init . --apply
```

Confirm the three files landed and the edge file is well-formed:

```bash
cat instrumentation.edge.ts         # registerOTel present, service name substituted
grep -q "registerOTel" instrumentation.edge.ts && echo "edge OK"
grep '"@vercel/otel"' package.json  # dep added
```

Sanity checks — the edge file must **not** carry any Node-only surface (Next's
edge bundler rejects `node:fs`, and it would defeat the point):

```bash
grep -n "node:fs\|neat-out\|NodeSDK\|BatchSpanProcessor" instrumentation.edge.ts \
  && echo "PROBLEM: edge file carries Node-only surface" \
  || echo "edge file is clean"
```

Then sync the lockfile (NEAT never touches lockfiles — that is yours):

```bash
npm install                          # pulls @vercel/otel + the OTel deps
npm run build                        # confirm the edge bundle compiles
```

If `npm run build` fails complaining that an edge file imports a Node API, the
failure is in your own edge routes, not the generated file — the generated file
imports only `@vercel/otel`, which ships an `edge-light` build Next selects
automatically.

## 2. Deploy to Vercel and set the OTLP env

Set the exporter env on the Vercel project so the deployed isolate exports to your
reachable daemon instead of the `localhost` default. These are the **same** vars
the generated file reads; setting them at the platform wins over the inline
`||=` defaults.

```bash
# The publicly reachable OTLP /v1/traces route of your NEAT daemon:
vercel env add OTEL_EXPORTER_OTLP_TRACES_ENDPOINT   production
#   e.g. https://neat.example.com/v1/traces   (or your tunnel URL + /v1/traces)

vercel env add OTEL_EXPORTER_OTLP_PROTOCOL          production
#   http/json

# Only if the daemon's OTLP receiver requires a token (NEAT_OTEL_TOKEN / NEAT_AUTH_TOKEN):
vercel env add NEAT_OTEL_TOKEN                       production
#   <the daemon's OTLP secret>
```

Then deploy:

```bash
vercel --prod
```

> The endpoint value must be the full `…/v1/traces` route. `@vercel/otel` uses
> `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` verbatim (it only appends `/v1/traces` to
> the *base* `OTEL_EXPORTER_OTLP_ENDPOINT`, which NEAT does not set).

## 3. Drive real edge traffic

Hit a surface that runs on the edge runtime. Middleware runs on **every** matched
request, so a handful of requests to any page it matches is enough:

```bash
for i in $(seq 1 20); do curl -s -o /dev/null "https://<your-app>.vercel.app/"; done
```

If you have an `export const runtime = 'edge'` route, hit that specifically:

```bash
for i in $(seq 1 20); do curl -s -o /dev/null "https://<your-app>.vercel.app/api/edge-thing"; done
```

Each edge invocation is a short-lived isolate. `@vercel/otel` flushes its spans
per-invocation by holding the invocation open with `waitUntil` until the root
span ends and the batch is exported — this is exactly why the generated file uses
`registerOTel()` and not a raw `BatchSpanProcessor` (whose timer-based flush loses
spans when the isolate is torn down mid-interval). You do not need to add any
flush code.

## 4. Confirm spans reached NEAT and minted OBSERVED edges

On the daemon side:

```bash
# The daemon logs OTLP receipts; edge spans arrive with your service.name.
neat status                          # daemon up, OTLP port listening

# Query the graph for OBSERVED edges carrying the edge service name.
# Edge middleware / handlers show up as SERVER spans on the service node,
# and any fetch() they make becomes an OBSERVED CLIENT edge.
neat query 'observed edges where service = "<service-name>"'   # or via the REST/MCP surface
```

Success looks like: the service node exists, and edges you know only the edge
runtime exercises (middleware → downstream fetch, an edge route → an upstream API)
carry `OBSERVED` provenance with a recent `lastObserved`. That is the whole point
of ADR-126 — the edge half of the app is no longer dark.

If you also have Node-runtime routes, drive those too and confirm **both**
runtimes report: Node routes via `instrumentation.node.ts`, edge routes/middleware
via `instrumentation.edge.ts`. Two runtimes, one service node, one receiver.

## Rollback / removal

The generated files are removable. On a **partial apply failure**, the installer
writes `neat-rollback.patch` at the package root listing every file it restored or
removed (including `instrumentation.edge.ts`) — the apply is transactional, so a
failure never leaves a half-instrumented tree.

To back the instrumentation out by hand, delete the three `instrumentation*.ts`
files, drop `@vercel/otel` and the three `@opentelemetry/*` deps from
`package.json`, and re-run your package manager's install. Deleting only
`instrumentation.edge.ts` and re-running `neat init --apply` regenerates exactly
that file (the deps and the Node pair are left alone) — the apply is idempotent
and file-presence-driven.

## Troubleshooting

**No edge spans at NEAT, Node spans fine.** Almost always the OTLP endpoint. The
inline default is `http://localhost:4318/v1/traces`; from a deployed Vercel
isolate `localhost` is the isolate itself, so spans go nowhere. Confirm
`OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` is set on the Vercel project to a
network-reachable `…/v1/traces` URL and that you redeployed after setting it
(env changes need a new deployment).

**Nothing exports at all, from either runtime.** Turn on `@vercel/otel`'s diag
logging with `OTEL_LOG_LEVEL=debug` (Vercel env) and read the function logs —
`registerOTel` logs its chosen exporter, protocol, and header names at startup.

**Edge spans go to Vercel's collector, not NEAT.** `@vercel/otel` has two
higher-precedence paths that *bypass* the env-configured OTLP exporter:
- If the deployment has **Vercel trace drains** configured (an OTel Collector
  integration on the project), `@vercel/otel` routes to the drain and skips the
  automatic env exporter. This is by design — and it is exactly the push-shaped
  path the (out-of-scope) Vercel Drains connector would receive. The env-based
  edge installer and a Vercel drain are mutually exclusive; pick one.
- If Vercel sets `VERCEL_OTEL_ENDPOINTS` (its internal collector), the same
  bypass applies.
  For the ADR-126 path to reach NEAT, the project must **not** have a competing
  Vercel trace drain configured.

**`tsc` in the host project complains about `@vercel/otel` types.** Run
`npm install` first — the dep is added to `package.json` but the lockfile and
`node_modules` are yours to sync (NEAT never writes lockfiles).

**Verifying without a live deploy.** The complete wire path — which env vars
`@vercel/otel@2.1.3` reads, that `http/json` selects its JSON exporter, that
`OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` becomes the exporter URL verbatim, and that
`waitUntil` drives per-invocation flush — is asserted against the generated file
in the `§7 Next.js edge-runtime branch (ADR-126)` suite in
`packages/core/test/audits/contracts.test.ts`. Those lock the contract this
runbook exercises live.
